# Vachanamrut RAG

A bilingual retrieval-augmented generation (RAG) system for querying two Swaminarayan scriptures:

- The Vachanamrut — 273 sacred discourses of Bhagwan Swaminarayan
- Swamini Vaato — 1,484 Vats across 7 Prakarans

Ask questions in English or Gujarati and receive answers grounded strictly in retrieved passages, with source citations.

---

## What it does

- Accepts questions in **English or Gujarati** and auto-detects the language
- Retrieves the most relevant passages from the **Vachanamrut**, **Swamini Vaato**, or both
- Generates answers in **both languages simultaneously** via an LLM
- Returns citations back to the exact discourse(s) or Vats the answer draws from
- Warns the user when no strongly relevant passages were found
- Maintains **per-session conversation history** for contextual follow-up questions
- Supports **Search** mode for fresh retrieval and **Continue conversation** mode for follow-up discussion from already-retrieved passages

## Setup

### 1. Clone and create a virtual environment

```bash
git clone <repo-url>
cd vachanmrut_rag
python3 -m venv venv
source venv/bin/activate
```

### 2. Install dependencies

```bash
pip install \
  flask flask-cors \
  chromadb \
  FlagEmbedding \
  transformers==4.51.3 \
  google-genai \
  python-dotenv \
  requests \
  beautifulsoup4 \
  numpy
```

> **Important:** `transformers` must be pinned to `4.51.3`. Newer versions break the `FlagEmbedding` import due to an internal API change.

### 3. Configure environment variables

```bash
cp .env.example .env
```

Edit `.env` and add your Gemini API key (required for `prod` LLM mode and for optional summary generation):

```
GEMINI_API_KEY=your_key_here
```

### 4. Update the root path in `config.py`

Open `config.py` and set `ROOT` to your local project directory:

```python
ROOT = Path("/your/path/to/vachanmrut_rag")
```

### 5. Pull the Ollama model (local mode only)

```bash
ollama pull qwen3:8b
```

Skip this if you plan to use Gemini (`prod` mode) exclusively.

## Building the Data and Index

Run these phases once, in order, before starting the app.

### Phase 1a — Vachanamrut Scraping

Fetches all 273 discourses from [anirdesh.com](https://www.anirdesh.com/vachanamrut) in both Gujarati and English. Saves structured records to `vachanamrut_data/vachanamrut_master.json` and individual `.txt` files per discourse. A 1.2s delay between requests is built in for polite scraping. Expected runtime: ~11 minutes.

```bash
python scripts/scrape_vachanamrut.py
```

The scraper is **resumable** — if interrupted, re-running it skips already-scraped discourses.

The output structure looks like this:

```
vachanamrut_data/
├── Gadhada_Pratham/
│   ├── 01_Gadhada_Pratham_1/
│   │   ├── gujarati.txt       # full Gujarati text
│   │   ├── english.txt        # full English translation
│   │   └── combined.txt       # both together
│   └── ...
└── vachanamrut_master.json
```

Each record in `vachanamrut_master.json` has this shape:

```json
{
  "1": {
    "vachno": 1,
    "section_en": "Gadhada_Pratham",
    "section_gu": "ગઢડા પ્રથમ",
    "num_in_section": 1,
    "title_en": "Unbroken Vritti",
    "title_gu": "અખંડ વૃત્તિનું",
    "body_en": "...",
    "body_gu": "...",
    "footnotes_en": "...",
    "footnotes_gu": "..."
  }
}
```

### Phase 1b — Swamini Vaato Scraping

Fetches all 1,484 Vats across 7 Prakarans from [anirdesh.com](https://www.anirdesh.com). Saves the complete bilingual dataset to `swamini_vaato_data/swamini_vaato_master.json`. The scraper is resumable, uses a built-in 1.2s delay between requests, and stores `body_en = ""` when a translation is unavailable.

```bash
python3 scripts/scrape_swamini_vaato.py
```

Prakaran counts:

- Prakaran 1: 342
- Prakaran 2: 192
- Prakaran 3: 74
- Prakaran 4: 140
- Prakaran 5: 407
- Prakaran 6: 295
- Prakaran 7: 34

### Phase 2a — Summary Generation *(optional)*

Generates a dense 200–300 word English summary for each discourse using Gemini. These are stored in the master JSON but are **not required** by the current retrieval pipeline, which works directly from passage text. Skip this unless you specifically need the summaries.

```bash
# Always test on a single discourse first
python scripts/generate_summaries.py --test --vachno 5

# Then run the full batch
python scripts/generate_summaries.py
```

Requires `GEMINI_API_KEY`. The script is **resumable** and includes exponential backoff for Gemini's free-tier rate limits (14 req/min).

### Phase 2b — Index Building

Splits discourses and Vats into natural paragraphs, merges any paragraph under 80 words into the next to avoid low-signal fragments, embeds each passage with BGE-M3, and stores the vectors in one persistent ChromaDB collection. Downloads the BGE-M3 model (~570MB) on first run.

```bash
python3 scripts/build_index.py --source both
```

Source options:

```bash
python3 scripts/build_index.py --source vachanamrut
python3 scripts/build_index.py --source swamini_vaato
python3 scripts/build_index.py --source both
```

Behavior:

- `--source both` rebuilds the entire unified collection from scratch
- `--source vachanamrut` rebuilds the collection from scratch with Vachanamrut data
- `--source swamini_vaato` adds Swamini Vaato passages into the existing collection

Expected output after a full rebuild: roughly 3,300 passage vectors total.

All indexed passages include a `source` field in metadata, so retrieval and citation building can distinguish between the two corpora. Vachanamrut passages are embedded with the discourse title prepended to the English text to improve title-based retrieval.

## Query Pipeline

```
User question
     │
     ▼
[1] Detect language (Gujarati if >20% of characters are in Gujarati Unicode range)
     │
     ▼
[2] Search mode:
    direct discourse lookup first
    - Vachanamrut section/number match, e.g. "Gadhada Pratham 16", "GP 16"
    - Swamini Vaato match, e.g. "Prakaran 2 Vat 30", "P2 V30"
    - fuzzy Vachanamrut title match
     │
     ▼
[3] If no direct match → embed query with BGE-M3 and run ChromaDB cosine search
     │
     ▼
[4] Search mode:
    prepend recent history for context, then send passages to the LLM
    
[4b] Continue mode:
     skip lookup and retrieval entirely, reuse the previous search passages,
     and answer from recent conversation history only
     │
     ▼
[5] LLM returns JSON with answer_en + answer_gu in a single call
     │
     ▼
[6] Deduplicate citations by discourse or Vat, return answer + sources to UI
```

## Web UI Features

- **Bilingual answers** — every response is generated in English and Gujarati simultaneously
- **Language toggle** — switch between English and Gujarati answers instantly
- **Corpus filters** — search the Vachanamrut, Swamini Vaato, or both
- **Search / Continue mode** — either retrieve fresh passages or continue discussing the last retrieved set
- **Citations** — every answer shows which discourse(s) or Vats it draws from
- **Low relevance warning** — shown when no strongly matching passages were found
- **Multi-turn memory** — follow-up questions are aware of previous turns within a session
- **New session** — clears conversation history with a confirmation prompt
- **Dark / light theme** — toggle in the header

## Design Decisions

**Why a unified passage index instead of separate corpus-specific stores?**
Both scriptures can now be searched through one ChromaDB collection with a `source` metadata field. This keeps retrieval logic simple while still allowing corpus filters, source-aware citations, and mixed search results when both corpora are enabled.

**Why passage-level indexing instead of summary-level?**
The initial design used Gemini-generated summaries as the retrieval unit — one vector per discourse. This produced loose matches because summaries describe overall themes, not specific content. The current design indexes individual passages directly across both corpora, so queries match against the actual scripture text rather than a paraphrase.

**Why BGE-M3?**
BGE-M3 is natively multilingual and handles Gujarati script without fine-tuning. A Gujarati question can retrieve passages indexed via their English text, and vice versa. It also runs efficiently on Apple Silicon via MPS.

**Why paragraph splitting over sentence splitting?**
Sentence-level splits lose context — a teaching mid-explanation is meaningless without the sentence before it. Splitting on double newlines (the natural paragraph boundary in the scraped text) preserves each teaching as a logical unit. Short paragraphs (Sanskrit verse lines, transitional phrases) are merged into the next paragraph to prevent low-signal fragments.

**Why Search and Continue as separate modes?**
Search mode is best when the user wants fresh retrieval from scripture. Continue mode is best when the user is asking a follow-up about passages already retrieved and does not want a new ChromaDB search to shift the context.

**Why a single LLM call for both languages?**
Generating English and Gujarati answers in one API call halves cost and latency. The LLM returns a structured JSON object with `answer_en` and `answer_gu` keys. The parser strips markdown fences and `<think>` blocks (present when using Qwen3 with thinking mode) before JSON parsing.

**Why is session history in-memory only?**
Persisting sessions to disk introduces complexity and privacy considerations without clear benefit. Each browser tab generates a new `session_id` and is an independent session. History is intentionally cleared on server restart.

## Known Constraints

- `transformers` must be pinned to `4.51.3` — later versions break `FlagEmbedding`
- The web server must run with a **single worker** — BGE-M3 is loaded once per process and is not safe to share across forked workers
- Gemini free tier is limited to 14 requests/minute — always use `--test --vachno N` before running the full summary batch
- Session history does not survive server restarts — by design
- `vachanamrut_chroma/`, scraped master JSON files, and the BGE-M3 model cache are not included in the repo and must be generated locally by running the setup phases
