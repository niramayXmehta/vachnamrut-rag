# Vachanamrut RAG

A bilingual retrieval-augmented generation (RAG) system for querying the Vachanamrut вЂ” the 273 sacred discourses of Bhagwan Swaminarayan, recorded between 1819 and 1829. Ask questions in English or Gujarati and receive answers grounded strictly in the scripture, with source citations.

---

## What it does

- Accepts questions in **English or Gujarati** and auto-detects the language
- Retrieves the most relevant passages from all 273 discourses using semantic search
- Generates answers in **both languages simultaneously** via an LLM
- Returns citations back to the exact discourse(s) the answer draws from
- Warns the user when no strongly relevant passages were found
- Maintains **per-session conversation history** for contextual follow-up questions

## Setup

### 1. Clone and create a virtual environment

```bash
git clone <repo-url>
cd vachanmrut_rag
python3 -m venv venv
source venv/bin/activate
```

### 2. Install dependencies

```bash
pip install \
  flask flask-cors \
  chromadb \
  FlagEmbedding \
  transformers==4.51.3 \
  google-generativeai \
  python-dotenv \
  requests \
  beautifulsoup4 \
  numpy
```

> **Important:** `transformers` must be pinned to `4.51.3`. Newer versions break the `FlagEmbedding` import due to an internal API change.

### 3. Configure environment variables

```bash
cp .env.example .env
```

Edit `.env` and add your Gemini API key (required for `prod` LLM mode and for Phase 2a):

```
GEMINI_API_KEY=your_key_here
```

### 4. Update the root path in `config.py`

Open `config.py` and set `ROOT` to your local project directory:

```python
ROOT = Path("/your/path/to/vachanmrut_rag")
```

### 5. Pull the Ollama model (local mode only)

```bash
ollama pull qwen3:8b
```

Skip this if you plan to use Gemini (`prod` mode) exclusively.

## Building the Data and Index

Run these phases once, in order, before starting the app.

### Phase 1 вЂ” Scraping

Fetches all 273 discourses from [anirdesh.com](https://www.anirdesh.com/vachanamrut) in both Gujarati and English. Saves structured records to `vachanamrut_data/vachanamrut_master.json` and individual `.txt` files per discourse. A 1.2s delay between requests is built in for polite scraping. Expected runtime: ~11 minutes.

```bash
python scripts/scrape_vachanamrut.py
```

The scraper is **resumable** вЂ” if interrupted, re-running it skips already-scraped discourses.

### Phase 2a вЂ” Summary Generation *(optional)*

Generates a dense 200вЂ“300 word English summary for each discourse using Gemini. These are stored in the master JSON but are **not required** by the current retrieval pipeline, which works directly from passage text. Skip this unless you specifically need the summaries.

```bash
# Always test on a single discourse first
python scripts/generate_summaries.py --test --vachno 5

# Then run the full batch
python scripts/generate_summaries.py
```

Requires `GEMINI_API_KEY`. The script is **resumable** and includes exponential backoff for Gemini's free-tier rate limits (14 req/min).

### Phase 2b вЂ” Index Building

Splits every discourse into natural paragraphs, merges any paragraph under 80 words into the next to avoid low-signal fragments, embeds each passage with BGE-M3, and stores the vectors in a persistent ChromaDB collection. Downloads the BGE-M3 model (~570MB) on first run.

```bash
python scripts/build_index.py
```

Expected output: ~1,800 passage vectors. Subsequent runs delete and fully rebuild the collection.

## Web UI Features

- **Bilingual answers** вЂ” every response is generated in English and Gujarati simultaneously

## Design Decisions

**Why passage-level indexing instead of summary-level?**
The initial design used Gemini-generated summaries as the retrieval unit вЂ” one vector per discourse. This produced loose matches because summaries describe overall themes, not specific content. The final design indexes ~1,800 individual passages directly, so queries match against the actual scripture text rather than a paraphrase.

**Why BGE-M3?**
BGE-M3 is natively multilingual and handles Gujarati script without fine-tuning. A Gujarati question can retrieve passages indexed via their English text, and vice versa. It also runs efficiently on Apple Silicon via MPS.

**Why paragraph splitting over sentence splitting?**
Sentence-level splits lose context вЂ” a teaching mid-explanation is meaningless without the sentence before it. Splitting on double newlines (the natural paragraph boundary in the scraped text) preserves each teaching as a logical unit. Short paragraphs (Sanskrit verse lines, transitional phrases) are merged into the next paragraph to prevent low-signal fragments.

**Why a single LLM call for both languages?**
Generating English and Gujarati answers in one API call halves cost and latency. The LLM returns a structured JSON object with `answer_en` and `answer_gu` keys. The parser strips markdown fences and `<think>` blocks (present when using Qwen3 with thinking mode) before JSON parsing.

**Why is session history in-memory only?**
Persisting sessions to disk introduces complexity and privacy considerations without clear benefit. Each browser tab generates a new `session_id` and is an independent session. History is intentionally cleared on server restart.

## Known Constraints

- `transformers` must be pinned to `4.51.3` вЂ” later versions break `FlagEmbedding`
- The web server must run with a **single worker** вЂ” BGE-M3 is loaded once per process and is not safe to share across forked workers
- Gemini free tier is limited to 14 requests/minute вЂ” always use `--test --vachno N` before running the full summary batch
- Session history does not survive server restarts вЂ” by design
- `vachanamrut_chroma/` and the BGE-M3 model cache are not included in the repo and must be generated locally by running the setup phases
